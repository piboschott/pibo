## Was gut funktioniert

- `test/mcp-cli.test.mjs` deckt die wichtigsten agent-facing CLI-Einstiege ab: kompakte Root-/Config-/Tools-Hilfe, `pibo mcp --version`, progressive MCP-Hilfe, `config init/add/show/describe/remove` und leere Registry.
- `test/mcp-agent-context.test.mjs` ist ein gutes kleines Subset für die Laufzeitwirkung: Es prüft, dass der MCP-Katalog lokale Config-Metadaten ohne Server-Verbindung liest und dass nur ausgewählte, beschriebene Server in den Agent-Kontext gelangen.
- Der Web-Kanal-Test verbindet mehrere Produktgrenzen sinnvoll: Katalogausgabe, Beschreibung per PATCH, Schreibwirkung auf `mcp_servers.json` und Persistenz der Agent-Auswahl.
- Die Tests verwenden temporäre Config-Dateien und setzen `MCP_CONFIG_PATH` gezielt. Das reduziert Kopplung an lokale Entwicklerumgebungen.
- Die Granularität ist überwiegend gut: MCP-CLI und MCP-Agent-Kontext laufen in unter zwei Sekunden und eignen sich als schnelles Entwicklungs-Subset.

## Schwächen und Risiken

1. **Validierungsfehler sind für MCP-Konfiguration nur dünn abgesichert.**  
   `src/mcp/config-command.ts` validiert fehlerhaftes JSON, nicht-Objekte, fehlende `command`/`url`-Felder und gegenseitig ausgeschlossene `command`+`url`. Die Tests prüfen aktuell fast nur den Happy Path von `config add`. Ein Bruch der Fehlertexte oder Exit-Pfade würde erst in manueller Nutzung auffallen.

2. **Beschreibungslimits sind doppelt implementiert, aber nicht gemeinsam contract-getestet.**  
   `src/mcp/agent-context.ts` und `src/apps/chat/web-app.ts` normalisieren MCP-Beschreibungen jeweils mit Whitespace-Trim und 480-Zeichen-Limit. Es gibt einen erfolgreichen PATCH-Test, aber keine gezielten Tests für leere, zu lange oder nicht-string Werte. Risiko: CLI und Web können auseinanderlaufen.

3. **Registry-Tests sind an den aktuellen Leerzustand gekoppelt.**  
   `src/mcp/registry.ts` enthält derzeit `const REGISTRY: RegistryEntry[] = []`, und die Tests erwarten explizit `No registry entries are currently bundled`. Sobald ein Preset ergänzt wird, werden diese Tests brechen, obwohl das Feature korrekt erweitert wurde. Das ist ein absichtlicher Guard, aber als langfristiger Test ist er fragil.

4. **Auswahl unbekannter MCP-Server wird nicht sichtbar getestet.**  
   `getMcpAgentContextFile()` ignoriert unbekannte oder unbeschriebene ausgewählte Server. Das ist wahrscheinlich gewollt, wird aber nur für `missing-description` explizit geprüft. Ein Test für einen wirklich unbekannten Namen würde die beabsichtigte Toleranz klarer machen.

5. **Custom-Agent-Profiltest prüft MCP-Weitergabe nicht direkt.**  
   `src/apps/chat/agent-profiles.ts` ruft `.withMcpServers(agent.mcpServers)` auf. `test/agent-profiles.test.mjs` nutzt `mcpServers: []`, sodass ein versehentliches Entfernen dieser Zeile nicht durch diesen kleinen Profiltest auffallen würde. Der Laufzeit-Kontext-Test deckt nur `InitialSessionContextBuilder` direkt ab, nicht den Custom-Agent-Adapter.

6. **Web-Test ist fachlich breit.**  
   Der MCP-Web-Test prüft Katalog, PATCH, Config-Persistenz und Agent-Erstellung in einem Test. Das ist wertvoll als Integrationstest, aber für Entwicklung an nur einem Teilbereich etwas grob. Kleine HTTP-Negativtests könnten schneller isolieren, ob Normalisierung, Routing oder Store-Persistenz gebrochen ist.

## Fehlende oder anzupassende Tests

Empfohlene kleine Ergänzungen:

1. **MCP-Config-Negativtests in `test/mcp-cli.test.mjs`:**
   - `config add demo '{bad json}'` liefert `CONFIG_INVALID_SERVER_JSON`.
   - `config add demo '[]'` oder `'null'` liefert `CONFIG_INVALID_SERVER`.
   - `config add demo '{"args":[]}'` verlangt `command` oder `url`.
   - `config add demo '{"command":"node","url":"https://example.com/mcp"}'` lehnt Mischkonfiguration ab.

2. **Gemeinsame Beschreibungskontrakte:**
   - Direkter Unit-Test für `normalizeMcpServerDescription()` mit Whitespace-Komprimierung, leerem Text und >480 Zeichen.
   - Web-HTTP-Test per `--test-name-pattern` für PATCH mit leerer, zu langer und nicht-string `description`.
   - Wenn möglich eine gemeinsame Konstante oder zumindest ein Test, der CLI- und Web-Limit synchron hält.

3. **Custom-Agent-MCP-Weitergabe:**
   - In `test/agent-profiles.test.mjs` einen kleinen Test ergänzen: Agent mit `mcpServers: ["filesystem"]` erzeugen und prüfen, dass `profile.create(...).mcpServers` diese Auswahl enthält.

4. **Unbekannte ausgewählte Server explizit festschreiben:**
   - In `test/mcp-agent-context.test.mjs` `getMcpAgentContextFile(["unknown", "filesystem"], configPath)` prüfen. Erwartung: `filesystem` erscheint, `unknown` nicht.

5. **Registry-Erwartung weniger fragil machen:**
   - Den Leerzustands-Test entweder als bewussten Sentinel kommentieren oder so umstellen, dass bei vorhandenen Presets Struktur und Hilfetext geprüft werden. Sonst blockiert der erste echte Registry-Eintrag ohne fachlichen Fehler.
