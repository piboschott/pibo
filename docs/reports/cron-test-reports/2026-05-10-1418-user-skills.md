# Test-Review: User Skills Store, Installer und CLI

**Lauf:** 2026-05-10 14:18 Europe/Berlin  
**Bereich:** User-Skill-Verwaltung: `src/user-skills/*`, `src/skills/cli.ts`, zugehörige Tests in `test/user-skills.test.mjs`.

## Kurzfazit

Die vorhandenen User-Skill-Tests sind schnell, isoliert und treffen einen wichtigen jüngeren Designpunkt: Skill-Beschreibungen leben in `SKILL.md`-Frontmatter statt im JSON-Store. Für die Breite des Features ist die Abdeckung aber zu schmal. Store-Basisfälle sind nur teilweise getestet; Installer, CLI-Verhalten, Rename/Delete/Enable-Flows und Fehlerpfade fehlen fast vollständig. Dieser Bereich eignet sich gut für mehrere sehr kleine Node-Test-Subsets statt für einen breiten Integrationslauf.

## Stärken der vorhandenen Tests

1. **Sehr granularer Store-Subset:** `node --test test/user-skills.test.mjs` ist ein gutes Entwicklungs-Subset für schnelle Änderungen an `src/user-skills/store.ts`.
2. **Temporäre Workspaces:** Die Tests nutzen `mkdtempSync(...)` und vermeiden damit Kopplung an echte `.pibo`-Daten.
3. **Wichtige Datenmodell-Regel abgedeckt:** Beide Tests prüfen, dass `description` aus `SKILL.md` gelesen und nicht dauerhaft im JSON-Store gespeichert wird.
4. **Regression gegen doppeltes Frontmatter:** Der Update-Test prüft, dass Markdown mit vorhandener Frontmatter nicht zu mehrfachen `---`-Blöcken führt.

## Schwächen und Risiken

1. **Tests importieren `dist` statt `src`:** `test/user-skills.test.mjs` importiert `../dist/user-skills/store.js`. Das ist für `npm test` nach dem Build korrekt, aber als granularer Entwicklungsbefehl kann ein veraltetes `dist/` falsche Sicherheit geben. Für schnelle TDD-Schleifen wäre ein zusätzliches `tsx`-/Source-nahes Testmuster oder ein klar dokumentierter Vorab-Build sinnvoll.
2. **Store-Flows sind unterabgedeckt:** Keine Tests für `deleteUserSkill`, `setUserSkillEnabled`, Rename mit Verzeichnisverschiebung, doppelte Namen, ungültige Namen, fehlende `SKILL.md`, ungültigen Store oder Sortierung.
3. **Installer komplett ungetestet:** `src/user-skills/installer.ts` enthält URL-Parsing, GitHub-API-Aufrufe, rekursive Downloads, Cleanup bei Fehlern und Deduplikation. Diese Logik ist deutlich risikoreicher als der aktuell getestete Frontmatter-Pfad.
4. **CLI nicht automatisiert geprüft:** `src/skills/cli.ts` nutzt `os.homedir()` und Commander-Ausgaben. Es gibt keinen Test, der `add/list/show/enable/disable/remove` in einem isolierten Home-Verzeichnis prüft.
5. **Parser-Kontrakt nicht klar fixiert:** `parseSkillMd` und `parseSimpleYaml` sind bewusst simpel. Tests sollten festlegen, welche Inputs unterstützt werden: kein Frontmatter, leere Description, Doppelpunkte in Description, unvollständige Frontmatter, `---` im Body.

## Fehlende oder anzupassende Tests

Empfohlene kleine Testgruppen:

1. **Store Lifecycle, ohne Netzwerk**
   - create → list → find → update description → enable/disable → delete
   - Rename `old-name` → `new-name` inklusive Pfad-/Verzeichnisprüfung
   - doppelte Namen und ungültige Namen (`UpperCase`, leer, zu lang)

2. **Frontmatter-/Parser-Kontrakt**
   - `parseSkillMd("plain body")`
   - Frontmatter mit `description: Text: with colon`
   - kaputte Frontmatter ohne Enddelimiter
   - Body mit `---`, damit die gewünschte Delimiter-Semantik explizit wird

3. **Installer mit gemocktem `fetch`**
   - `parseSkillUrl` für GitHub-Repo, GitHub-Tree, Shorthand und `skills.sh`
   - erfolgreicher Download eines Skill-Verzeichnisses mit Zusatzdateien
   - Cleanup bei Download-Fehler
   - Deduplikation, wenn `SKILL.md` einen Namen enthält, der schon im Store existiert

4. **CLI-Skill-Subset**
   - In einem temporären `HOME`/isolierten cwd `pibo skills add --file ...`, `list`, `show`, `disable`, `enable`, `remove` prüfen.
   - Ziel: sicherstellen, dass `src/skills/cli.ts` nicht nur Store-Funktionen aufruft, sondern Namen/IDs und Ausgaben korrekt verdrahtet.

## Empfohlene granulare Test-Kommandos

Aktuell sinnvoll:

```bash
node --test test/user-skills.test.mjs
npm run dev -- skills --help
```

Nach Ergänzung weiterer kleiner Tests empfehle ich getrennte Subsets statt eines großen Sammeltests:

```bash

```

Für spätere Integrations-/Deployment-Stufen reicht weiterhin:

```bash

```

Das sollte aber nicht der primäre Entwicklungsloop für User-Skill-Änderungen sein, weil Build und Web-UI-Bundles dafür zu breit sind.

4. Dokumentieren oder ändern, wie granulare Tests gegen `src` statt möglicherweise altes `dist` laufen sollen. Bis dahin bei Store-Änderungen vor dem Subset mindestens `npm run build` oder ein gezielter TypeScript-Build einplanen.
