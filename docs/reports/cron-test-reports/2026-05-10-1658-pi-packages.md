## Was gut funktioniert

1. **Gutes Core-Subset für lokale Entwicklung**  
   `test/pi-packages.test.mjs` ist ein brauchbares schnelles Subset für Parser, Store, Runtime-Brücke und CLI-Basisfluss. Der Test ist klein genug für gezielte Arbeit an `src/pi-packages/*`.

2. **Sinnvolle Product-Boundary-Trennung**  
   Die Tests unterscheiden globale Registrierung (`upsertPiPackage`, `listPiPackages`), Profile-Selektion (`InitialSessionContextBuilder.withPiPackages`) und Runtime-Aktivierung (`getPiPackageRuntimeOptions`). Das passt zur Spezifikation: Registrieren aktiviert Pakete nicht automatisch.

3. **Legacy-Kompatibilität wird konkret geprüft**  
   Der Test für alte Store-Einträge mit `installed: true` deckt ein wichtiges Migrationsrisiko ab: alte Daten werden als `installStatus: "installed"` und `enabled: true` gelesen.

4. **Web-Sicherheitsregel ist zumindest als Integration abgedeckt**  
   `test/web-channel.test.mjs` prüft, dass Browser-Adds keine lokalen Pfade akzeptieren. Das ist die richtige Richtung, weil die CLI lokale Pfade darf, die Web-API aber nicht.

5. **Progressive CLI-Discovery wird explizit getestet**  
   Der CLI-Test prüft, dass `pibo pi-packages --help` kompakt bleibt und nicht Store-Details ausgibt. Das stützt die Projektregel zur iterativen CLI-Discoverability.

## Schwächen und Risiken

1. **Tests importieren `dist`, nicht `src`**  
   Alle Pi-Package-Tests importieren aus `../dist/...`. Das ist konsistent mit vielen Projekt-Tests, birgt aber bei Einzeltestläufen ein Drift-Risiko: `node --test test/pi-packages.test.mjs` prüft den zuletzt gebauten Stand, nicht zwingend aktuelle TypeScript-Änderungen. Für schnelle Entwicklung ist das nützlich, aber als Analyse muss klar sein: Vor verbindlicher Bewertung ist mindestens ein gezielter Build oder Typecheck nötig.

2. **REQ-009 ist in der Spezifikation als implementiert markiert, aber nicht direkt getestet**  
   `docs/specs/capabilities/pi-packages.md` nennt „Failed refresh preserves a previous installed record“. `src/pi-packages/store.ts` enthält die Logik in `upsertPiPackage()`, aber `test/pi-packages.test.mjs` hat keinen Test, der einen installierten Datensatz per Error-Input aktualisiert und danach `installPath`, `installStatus`, `updatedAt` und gemergte Diagnostics prüft.

3. **Runtime-Fehlerpfade sind nur teilweise granular abgedeckt**  
   `test/pi-packages.test.mjs` prüft erfolgreiche Selektion, fehlende Selektion, unbekannte Selektion und global disabled. Es fehlen kleine Runtime-Tests für:
   - registriertes Paket mit `installStatus: "error"`,
   - registriertes Paket mit `installStatus: "registered"` ohne Runtime-Pfad,
   - installiertes Paket mit gelöschtem/nicht vorhandenem `installPath`,
   - doppelte ausgewählte Pakete mit gleichem `installPath` und erwarteter De-Duplizierung.

4. **CLI-Doctor und CLI-Fehlerausgaben sind nicht abgedeckt**  
   `src/pi-packages/cli.ts` implementiert `doctor`, Unknown-Inspect und Unknown-Remove. Diese Pfade sind gute Agenten-Operator-Flows, aber derzeit nicht im schnellen CLI-Subset enthalten.

5. **Web-Pi-Package-Test ist zu breit für Entwicklungs-Feedback**  
   `test/web-channel.test.mjs` verbindet Catalog, PATCH enable/disable, Agent-Erstellung und Delete-Konflikt in einem Integrationstest. Das ist wertvoll als später Flow-Test, aber für UI/API-Änderungen zu grob. Ein Fehler zeigt nicht schnell, ob Catalog-Invalidierung, Same-Origin-Check, Agent-Store-Validierung oder Delete-Guard gebrochen ist.

6. **Npm-Install-Fehler werden nicht isoliert getestet**  
   `installOrResolvePiPackage()` hat Timeout-/Fehler-/Missing-Output-Zweige. Die aktuellen Tests vermeiden echte npm-Pfade sinnvollerweise. Es fehlt aber ein mockbares oder extrahiertes Subset, das Installfehler ohne Netzwerk und ohne 120-s-Risiko prüft.

## Fehlende oder anzupassende Tests

### Kurzfristig sinnvolle neue Core-Tests in `test/pi-packages.test.mjs`

- `upsertPiPackage preserves previous installed package when refresh input is error`  
  Prüft konkret REQ-009 gegen `src/pi-packages/store.ts`.

- `pi package runtime reports skipped error, uninstalled, and missing-path packages`  
  Drei kleine Records in einem Temp-CWD; Assertions auf Diagnostics und leere `additionalExtensionPaths`.

- `pi package runtime de-duplicates selected runtime paths`  
  Zwei selektierte Records mit gleichem `installPath`; erwartet genau ein Pfad.

- `pibo pi-packages doctor reports invalid local source`  
  Store mit nicht mehr existierendem lokalen Pfad; `doctor` sollte `error` ausgeben, ohne einen Web-/Gateway-Test zu brauchen.

### Mittelfristig Web-Subsets aus `test/web-channel.test.mjs` herauslösen

Empfehlung: ein kleineres Testmodul, z. B. `test/chat-pi-packages-api.test.mjs`, das nur `/api/chat/pi-packages` betrachtet:

- GET list requires auth.
- POST/PATCH/DELETE require same-origin JSON.
- POST/PATCH source rejects non-`pi.dev` sources.
- PATCH rejects non-boolean `enabled`.
- DELETE selected package returns 409 with affected agent names.

Der bestehende breite `web-channel`-Flow kann als Integrations-/Deployment-Suite bleiben, sollte aber nicht das einzige Feedback für diese API sein.
