## Was gut funktioniert

- `test/runs.test.mjs` ist thematisch gut abgegrenzt und schnell. Das ist ein gutes Entwicklungs-Subset für Änderungen an `src/runs/*` und den Router-Reminder-Flüssen.
- Die Kernzustände des In-Memory-Registrys sind sinnvoll abgedeckt: tracked vs. detached, wait timeout, owner dispose, prune, ack, wiederholte Turn-End-Reminder.
- Die Router-Tests prüfen nicht nur Registry-Methoden, sondern auch den wichtigen Produktfluss: Abschluss/Fehler eines Yielded Runs erzeugt eine kompakte Service-Nachricht an die Parent-Session.
- Der Test `router invalidates stale queued run notifications after read` deckt ein konkretes Fragilitätsrisiko ab: bereits gequeue-te Reminder dürfen nach `read` nicht stale bleiben.

## Schwächen und Risiken

1. **Store-backed Registry ist im Run-Test kaum abgedeckt.**  
   `PiboSessionRouter` nutzt bei persistenter Session eine `PiboRunRegistry` mit `PiboReliabilityStore`. `test/runs.test.mjs` prüft überwiegend die In-Memory-Variante (`new PiboRunRegistry()` bzw. Router mit `persistSession: false`). Die Persistenzpfade in `src/reliability/store.ts` (`createRun`, `updateRun`, `listRuns`, `recoverInterruptedRuns`, `pruneRuns`) werden nur indirekt und nicht als Registry-Verhalten getestet.

2. **Recovery-/Retry-Semantik ist riskant, aber nicht granular abgesichert.**  
   `recoverInterruptedRuns()` kann laufende Runs in `queued` oder `failed` überführen. `PiboRunStatus` erlaubt `queued`, die aktuellen Registry-Tests prüfen aber keine queued-Recovery und keine Rehydration aus dem Store. Das ist besonders relevant, weil Reminder- und Listenlogik queued Runs aktuell wie nicht-terminale Runs behandelt.

3. **Tool-Wrapper-Abdeckung ist einseitig.**  
   `createRunToolDefinitions()` wird nur für `pibo_run_start` geprüft. Für `pibo_run_list`, `pibo_run_status`, `pibo_run_wait`, `pibo_run_read`, `pibo_run_cancel` und `pibo_run_ack` fehlen kleine Contract-Tests, obwohl diese Tools agent-facing sind und ihre Texte/Details wichtig für die Bedienbarkeit sind.

4. **Fehlerfälle der Tool-Ausführung fehlen teilweise.**  
   `src/runs/tools.ts` behandelt `resultObject.isError === true` als Exception und extrahiert Text aus `content`. Dafür gibt es keinen direkten Tool-Wrapper-Test. Ein Bruch hier würde erst in breiteren Runtime-/Router-Flows auffallen.

5. **Tests laufen gegen `dist/`, nicht gegen `src/`.**  
   Das ist projektweit üblich, macht aber einzelne `node --test test/runs.test.mjs`-Läufe anfällig für stale Build-Artefakte. Für granulare Entwicklung sollte der empfohlene Subset-Befehl diesen Umstand explizit berücksichtigen.

## Fehlende oder anzupassende Tests

Empfohlene kleine Ergänzungen, ohne breite E2E-Suite:

1. **Store-backed Registry Contract** in `test/runs.test.mjs` oder separatem `test/run-registry-store.test.mjs`:
   - `new PiboRunRegistry({ store: new PiboReliabilityStore(':memory:') })`
   - Start eines tracked Runs, Complete, neue Registry mit demselben Store, `list/status/read` prüfen.
   - Prune über Registry prüfen und sicherstellen, dass Store-Zeilen verschwinden.

2. **Recovery-Fälle gezielt testen**:
   - Nicht-retryable running run ohne gültigen Claim wird bei Registry-Konstruktion `failed` und terminal.
   - Retryable running run wird `queued` und erscheint sinnvoll in `list/status`.
   - Prüfen, ob queued Runs Reminder erzeugen sollen oder nicht. Falls ja: Erwartung explizit testen; falls nein: Logik anpassen und testen.

3. **Tool-Definitionen einzeln testen**:
   - `pibo_run_list` reicht `includeConsumed/includeDetached` korrekt durch.
   - `pibo_run_wait` nutzt Default `30000` und formatiert Timeout als normalen Zustand.
   - `pibo_run_read` gibt bei terminalem Result nur den Resulttext zurück und liefert Details vollständig.
   - `pibo_run_cancel`/`pibo_run_ack` rufen Controller korrekt auf.
   - Unknown yieldable tool wirft `Unknown or non-yieldable tool`.
   - Yieldable Tool mit `{ isError: true }` wird als failed Run behandelt.

4. **Router-Reminder-Subsets ergänzen**:
   - `ackRun` entfernt stale queued Reminder analog zu `readRun`.
   - `cancelRun` entfernt stale queued Reminder und queued danach keinen bereits konsumierten Run neu.
   - Mehrere Owner-Sessions: Reminder und Listen bleiben strikt owner-isoliert.

## Empfohlene granulare Test-Kommandos

Für reine Run-Control-Änderungen nach einem aktuellen Build:

```bash
node --test test/runs.test.mjs
```

Wenn TypeScript-Quellen geändert wurden und keine Web-UI betroffen ist:

```bash
tsc -p tsconfig.json && node --test test/runs.test.mjs
```

Für Persistenz-/Job-Änderungen angrenzend an Runs:

```bash
tsc -p tsconfig.json && node --test test/runs.test.mjs test/reliability-store.test.mjs
```

Breitere Suites wie `npm test` sollten erst später im Flow laufen, weil sie `npm run build` inklusive Web-UI-Build ausführen und deutlich breiter sind.

## Konkrete nächste Schritte

1. Ein kleines store-backed Registry-Testset ergänzen, bevor an `src/runs/registry.ts` oder `src/reliability/store.ts` weitergearbeitet wird.
2. `queued`-Semantik für recovered Runs fachlich entscheiden und als Test festschreiben.
3. `createRunToolDefinitions()` mit Contract-Tests für jedes `pibo_run_*` Tool absichern.
4. In Entwicklerdokumentation oder Test-README klar markieren, dass direkte `node --test`-Subsets gegen `dist/` laufen und einen frischen Compile brauchen.
