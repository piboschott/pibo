# Test-Review: Data V2 Chat-Ingest

Datum: 2026-05-10 14:29 Europe/Berlin

## Untersuchter Bereich

Granularer Fokus dieses Laufs: `ChatDataIngestService` und die direkt darunterliegende Pibo-Data-V2-Persistenz.

Betrachtete Tests:

- `test/data-v2-ingest-service.test.mjs`
- `test/data-v2-store.test.mjs`

Betrachteter Code:

- `src/data/ingest-service.ts`
- `src/data/pibo-store.js` indirekt über die Dist-Imports der Tests
- `src/core/events.ts`
- `src/apps/chat/data/event-command-service.ts`
- Chat-Web-Aufrufstellen in `src/apps/chat/web-app.ts` per Suche

Ausgeführte Checks:

```bash
node --test test/data-v2-ingest-service.test.mjs
node --test test/data-v2-store.test.mjs test/data-v2-ingest-service.test.mjs
```

Ergebnis: Beide begrenzten Subsets liefen erfolgreich. Das kombinierte Subset hatte 9 bestandene Tests in ca. 346 ms.

## Was gut funktioniert

- Die Ingest-Tests sind klein und schnell. Sie eignen sich gut als Entwicklungs-Subset, weil sie keine Gateway-, Browser- oder Build-Suite starten.
- `test/data-v2-ingest-service.test.mjs` prüft zentrale Ingest-Verträge statt nur Implementierungsdetails:
  - idempotente User-Message-Aufnahme über `clientTxnId`,
  - Externalisierung großer User-Payloads,
  - idempotentes Shadowing von Assistant-Messages,
  - progressive Tool-Call-Argument-Snapshots,
  - Tool-Output-Observation-Mapping.
- `test/data-v2-store.test.mjs` ergänzt darunter die Basisannahmen der Store-Schicht: Schema-Idempotenz, Payload-Dedupe, Event-Log-Idempotenz und einfache Message-/Observation-Listen.
- Der Testaufbau mit temporären Stores und `:memory:` hält die Tests isoliert und vermeidet Abhängigkeiten von realen Gateway-Daten.
- Die Tests decken eine wichtige Produktgrenze ab: Chat-Web kann alte Event-Flüsse weiter bedienen, während `ChatDataIngestService` V2-Daten für Event-Log, Messages, Observations und Navigation shadowt.

## Schwächen und Risiken

1. **Dist-Import statt Source-Import**
   - Die Tests importieren aus `../dist/...`.
   - Das ist konsistent mit vielen bestehenden Tests, birgt aber ein Analyse-Risiko: Ein einzelner `node --test ...` prüft den zuletzt gebauten Stand, nicht zwingend den aktuellen Source-Stand.
   - Für schnelle lokale Subsets sollte klar dokumentiert sein, ob vorher `npm run build` nötig ist oder ob ein Source-Testmodus existieren soll.

2. **Viele Output-Event-Typen bleiben ungetestet**
   - `src/core/events.ts` definiert u. a. `message_queued`, `message_started`, `message_finished`, `assistant_delta`, `thinking_*`, `tool_execution_started`, `tool_execution_updated`, `subagent_session`, `execution_result`, `session_error`, `pi_event`, `compaction_start` und `compaction_end`.
   - `test/data-v2-ingest-service.test.mjs` deckt davon aktuell vor allem `assistant_message`, `tool_call` und `tool_execution_finished` ab.
   - Das Risiko liegt in den vielen Mapping-Funktionen in `src/data/ingest-service.ts`: `retentionClassForOutputEvent`, `payloadForOutputEvent`, `previewTextForOutputEvent`, `attributesForOutputEvent`, `observationKindForOutputEvent`, `observationStatusForOutputEvent` und `outputSessionStatus` können bei neuen Events regressieren, ohne dass ein Test anschlägt.

3. **Session-Navigation-Status ist nur teilweise geprüft**
   - Für User-Messages wird `lastMessagePreview` und `room_id` geprüft.
   - Für Output-Events wird Navigation nur indirekt bei Assistant-Messages geprüft; `message_finished -> idle` und `session_error -> error` sind im Ingest-Subset nicht abgedeckt.
   - Gerade diese Statuswerte sind UI-relevant, weil Chat-Web Session-Listen und Signal-/Read-State-Darstellung darauf aufbauen.

4. **Payload-Externalisierung ist nur für User-Text geprüft**
   - Große User-Texte sind abgedeckt.
   - Große Assistant-Texte, große Tool-JSON-Results und große `pi_event`-/`execution_result`-Payloads sind nicht im Ingest-Subset.
   - `writePayloadIfLarge` unterscheidet Text- und JSON-Schwellen; diese Verzweigung verdient gezielte Tests.

5. **Sequenz- und Duplicate-Kanten sind schmal abgedeckt**
   - Die User-Message-Idempotenz wird für `clientTxnId` getestet.
   - Nicht getestet ist der nicht-idempotente Pfad ohne `clientTxnId`: zwei gleiche Texte sollten zwei Messages/Event-Log-Zeilen erzeugen.
   - Ebenfalls nicht explizit geprüft: `sessionSequence` und Message-`sequence` bei gemischten User-/Assistant-/Tool-Events.

6. **Legacy-Doppelpfad bleibt fragil**
   - `src/apps/chat/data/event-command-service.ts` hat `appendOutputEvent` bewusst auf `undefined` reduziert, damit `ChatDataIngestService` Runtime-Output besitzt.
   - Es gibt keinen direkt benannten Test, der diesen Nicht-Doppelinsert-Vertrag zusammen mit Ingest absichert. Ein zukünftiger Rückbau könnte erneut doppelte `event_log`-Einträge erzeugen.

## Fehlende oder anzupassende Tests

Empfohlene kleine Ergänzungen, ohne breite Integrationssuite:

1. **Output-Event-Mapping-Matrix in `test/data-v2-ingest-service.test.mjs`**
   - Table-driven Test für repräsentative Events:
     - `message_finished` erzeugt Event/Observation und setzt Navigation auf `idle`.
     - `session_error` erzeugt Error-Observation und setzt Navigation auf `error`.
     - `thinking_delta` bekommt `kind: "thinking"` und Text-Payload.
     - `execution_result` bekommt `kind: "execution"` und JSON-Payload.
     - `compaction_start` ist `running`, `compaction_end` ist `completed` bzw. enthält Fehlerattribute.

2. **Payload-Schwellen für Output-Events**
   - Großer `assistant_message.text` externalisiert Text und setzt kein `inlineText`.
   - Großes `tool_execution_finished.result` externalisiert JSON und setzt kein `inlinePayload`.

3. **Sequenz-Regressionstest**
   - Ein Test mit User-Message, Assistant-Message und Tool-Output in einer Session sollte `sessionSequence` und Message-`sequence` prüfen.
   - Ziel: Schutz gegen Off-by-one-Fehler in `nextEventSequence` und `nextMessageSequence`.

4. **Doppelinsert-Guard für Chat-Event-Command-Service**
   - Kleiner Test gegen `ChatEventCommandService.appendOutputEvent(...)`, der bestätigt, dass persistierbarer Runtime-Output nicht zusätzlich von dieser Legacy-Schicht in `event_log` geschrieben wird.
   - Das passt zu `src/apps/chat/data/event-command-service.ts`, wo der Kommentar diesen Vertrag bereits beschreibt.

5. **Nicht-idempotenter User-Pfad**
   - Test ohne `clientTxnId`, der zwei gleiche User-Messages absendet und zwei verschiedene Message-IDs/Event-Zeilen erwartet.
   - Dadurch wird die bewusste Semantik zwischen Client-Retry und echter Wiederholung dokumentiert.

## Empfohlene granulare Test-Kommandos

Für schnelle Entwicklung an Data-V2-Store oder Ingest:

```bash
node --test test/data-v2-store.test.mjs test/data-v2-ingest-service.test.mjs
```

Für reine Ingest-Änderungen:

```bash
node --test test/data-v2-ingest-service.test.mjs
```

Für Änderungen an Chat-Web-Datenpfaden zusätzlich, aber noch ohne vollständige Build-/Browser-Suite:

```bash
node --test test/chat-data-v2-legacy-guard.test.mjs test/data-v2-store.test.mjs test/data-v2-ingest-service.test.mjs
```

Wichtig: Da diese Tests aus `dist/` importieren, sollte vor Review oder CI-ähnlicher Validierung ein Build laufen:

```bash
npm run build
node --test test/data-v2-store.test.mjs test/data-v2-ingest-service.test.mjs
```

Für die enge Entwicklungsiteration ist der einzelne `node --test`-Lauf nützlich, solange der gebaute Stand bewusst aktuell gehalten wird.

## Konkrete nächste Schritte

1. `test/data-v2-ingest-service.test.mjs` um eine kleine table-driven Mapping-Matrix für `message_finished`, `session_error`, `thinking_delta`, `execution_result` und `compaction_*` erweitern.
2. Einen gezielten Output-Payload-Test für große Assistant- und Tool-Payloads ergänzen.
3. Einen Sequenztest für gemischte Event-Typen in einer Session ergänzen.
4. Einen Guard-Test für `ChatEventCommandService.appendOutputEvent` ergänzen, damit Runtime-Output nicht doppelt im V2-Event-Log landet.
5. In der Testdokumentation oder im README der Tests klarer markieren, welche schnellen Subsets `dist/` voraussetzen.
