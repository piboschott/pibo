## Was bereits gut funktioniert

- `src/apps/chat/cron-api.ts` trennt öffentliche Routen klar: Status, Jobs, Runs, einzelne Job-Ressourcen und manuelles `run`.
- Mutierende Requests (`POST`, `PATCH`, `DELETE`, manuelles `run`) verlangen `application/json` und einen Same-Origin-`Origin`-Header. Das ist als Sicherheitsgrenze sinnvoll und testbar.
- Owner-Scope wird konsequent über `webSession.ownerScope` in Store-Operationen geführt: `listJobs`, `createJob`, `getOwnedJob`, `updateJob`, `removeJob`, `listRuns`.
- Room-Ziele verwenden `requireRoomAccess(..., "write")` und lehnen archivierte Räume ab. Personal-Ziele werden auf den aktuellen Principal begrenzt.
- Die UI-Client-Funktionen in `src/apps/chat-ui/src/api.ts` bilden die API kompakt ab (`getCronJobs`, `postCronJob`, `patchCronJob`, `deleteCronJob`, `runCronJobNow`, `getCronRuns`) und eignen sich gut als Referenz für API-Testfälle.
- `test/web-http.test.mjs` zeigt bereits ein leichtgewichtiges Muster für HTTP-nahe Tests ohne kompletten Gateway-Start.

## Schwächen und Risiken

1. **Keine dedizierte Cron-API-Testdatei gefunden.**  
   Die Cron-Domain hat Store-/CLI-Smoke-Tests, aber die webseitige Zugriffsschicht in `src/apps/chat/cron-api.ts` ist nicht direkt abgesichert. Gerade diese Schicht enthält sicherheitsrelevante Regeln: Same-Origin, Content-Type, Owner-Scope, Room-Access, Archivschutz und Profilauflösung.

2. **Security-Fehlerpfade sind aktuell nur Code-Review-verifiziert.**  
   Besonders wichtig wären Tests für fehlenden `Origin`, falschen `Origin`, falschen Content-Type, fremde Personal-Targets und archivierte Räume. Ohne Tests können kleine Refactors in `requireSameOriginJsonRequest()` oder `normalizeTarget()` unbemerkt Schutzlogik abschwächen.

3. **API-Fehlerklassifikation kann uneinheitlich werden.**  
   `normalizeSchedule()` übersetzt viele Fehler in `PiboWebHttpError(400)`, aber einige Validierungen passieren später im Store, z. B. bei direkt übergebenem `{ kind: "at", at: ... }` oder rohen Cron-Schedules. Falls die zentrale Web-Schicht generische Errors als 500 behandelt, könnten fehlerhafte Nutzereingaben falsche Statuscodes erzeugen. Das sollte mit API-Tests geklärt werden.

4. **Manuelles `run` hängt an globalem Service-State.**  
   `POST /api/chat/cron/jobs/:id/run` verwendet `getPiboCronService()`. Das ist produktseitig plausibel, aber im Test fragiler als reine Store-/Handler-Tests. Ohne einen kleinen Test bleibt unklar, ob ein nicht laufender Service zuverlässig 503 liefert und ob der Owner-Scope beim manuellen Start erhalten bleibt.

5. **Frontend-CronArea ist breit und derzeit schwer granular zu testen.**  
   `src/apps/chat-ui/src/CronArea.tsx` enthält Listen, Draft-State, Schedule-Konvertierung, Room-Auswahl, Speichern, Löschen und manuelles Starten in einer Komponente. Ohne UI-Testinfrastruktur ist das akzeptabel, aber reine Hilfsfunktionen wie Draft-zu-Input-Konvertierung sollten perspektivisch leichter isolierbar werden. In diesem Lauf wurde keine UI-Änderung empfohlen, nur die Testbarkeit bewertet.

## Fehlende oder anzupassende Tests

### 1. Neue schmale Suite: `test/chat-cron-api.test.mjs`

Empfohlene Testfälle mit Fake-`roomService`, Fake-`context` und `PiboCronStore({ path: ":memory:" })`:

- `POST /api/chat/cron/jobs` ohne `Origin` gibt 403.
- `POST /api/chat/cron/jobs` mit `text/plain` gibt 415.
- `POST /api/chat/cron/jobs` mit fremdem `target.principalId` gibt 403.
- Room-Target ruft `requireRoomAccess(roomId, currentPrincipal, "write")` auf.
- Archivierter Room wird mit 403 abgelehnt.
- Unbekanntes Profil wird mit 400 abgelehnt.
- `GET /api/chat/cron/runs?jobId=<fremderJob>` gibt 404 statt Runs zu leaken.
- `PATCH /api/chat/cron/jobs/:id` ohne Felder gibt 400.

### 2. Kleine Fehlerstatus-Tests für Schedule-Normalisierung

- Direktes `{ kind: "at", at: "not-a-date" }` sollte als Nutzerfehler behandelt werden, nicht als interner Fehler.
- Direktes `{ kind: "cron", expr: "bad" }` sollte 400 ergeben.
- `{ kind: "every", everyMs: 1000 }` sollte 400 ergeben.

Diese Tests klären, ob die API-Schicht Store-Validierungsfehler sauber in HTTP-Fehler übersetzt oder ob ein expliziter Wrapper nötig wäre.

### 3. Service-Integration separat halten

Für `src/cron/service.ts` nicht in dieselbe API-Suite gehen. Besser eine spätere, eigene `test/cron-service.test.mjs` mit Fake-ChannelContext:

- `runJobNow()` reserviert einen manuellen Run.
- Erfolgreiches `message_finished` markiert den Run als `ok`.
- `session_error` markiert den Run als `error`.
- Timeout ist mit kleinem `runTimeoutMs` deterministisch.

## Konkrete nächste Schritte

1. `test/chat-cron-api.test.mjs` als kleine Handler-Unit-Suite einführen. Kein echter Gateway-Start, keine Browser-Tests.
2. Dabei wiederverwendbare Test-Fixtures für `webSession`, `context.channelContext.getProfiles`, `roomService` und `:memory:`-CronStore anlegen.
3. Zuerst Security-/Access-Fälle testen, danach erfolgreiche `POST`/`PATCH`/`GET runs`-Flows.
4. Wenn Fehlerstatus-Tests zeigen, dass Store-Validierungsfehler als 500 enden, gezielt an `handleChatCronApiRequest()` oder der zentralen Web-Fehlerbehandlung nachbessern.
5. `CronArea` vorerst nicht breit per Browser testen; erst die API-Verträge stabilisieren. UI-E2E nur für einen Happy Path später im Deployment-Flow.

## Bewertung

Die Cron-API ist fachlich sinnvoll strukturiert, aber als Sicherheits- und Produktgrenze noch untertestet. Die beste nächste Investition ist keine große End-to-End-Suite, sondern eine sehr kleine Handler-Suite für `src/apps/chat/cron-api.ts`. Sie würde die wichtigsten Risiken mit wenigen, schnellen Tests abdecken und klare Entwicklungs-Subsets schaffen.
