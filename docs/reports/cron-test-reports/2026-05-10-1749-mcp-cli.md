# Test-Review: MCP-CLI-Konfiguration und Discovery

Datum: 2026-05-10 17:49 Europe/Berlin  
Bereich: `pibo mcp` CLI, MCP-Konfiguration, MCP-Agent-Kontext

## Untersuchter Bereich

Dieser Lauf betrachtet den MCP-CLI-Pfad als agentenorientierte Discovery-Fläche:

- CLI-Parsing und Dispatch in `src/mcp/index.ts`
- Konfigurationsbefehle in `src/mcp/config-command.ts` und `src/mcp/config.ts`
- Agent-Kontext-Erzeugung in `src/mcp/agent-context.ts`
- Discovery-/Runtime-nahe Commands in `src/mcp/commands/list.ts`, `src/mcp/commands/info.ts`, `src/mcp/commands/grep.ts`, `src/mcp/commands/call.ts`
- Bestehende Tests in `test/mcp-cli.test.mjs` und `test/mcp-agent-context.test.mjs`

Ausgeführt wurden nur die zwei passenden Node-Testdateien und ein kleiner manueller Negativcheck für `config add`.

## Stärken der vorhandenen Tests

1. **Sehr schnelle, agentenfreundliche CLI-Smoke-Suite**  
   `test/mcp-cli.test.mjs` prüft reale `dist/bin/pibo.js`-Aufrufe über `execFile`. Das ist für CLI-Verhalten wertvoller als reine Unit-Tests, bleibt aber mit rund einer Sekunde schnell genug für Entwicklungs-Feedback.

2. **Progressive Discovery ist direkt abgesichert**  
   Die Tests prüfen, dass Root-Help und `pibo mcp config help` kompakt bleiben und detaillierte Schema-/Example-Ausgabe erst hinter `pibo mcp config schema` erscheint. Das passt zur Projektregel der iterativen CLI-Discoverability.

3. **Konfigurations-Roundtrip deckt den wichtigsten Happy Path ab**  
   `init`, `add`, `show`, `describe` und `remove` laufen in einem temporären Verzeichnis gegen eine echte JSON-Datei. Dadurch sind JSON-Serialisierung, Dateipfade und Pibo-Metadaten gemeinsam geprüft.

4. **Agent-Kontext wird ohne Serververbindung getestet**  
   `test/mcp-agent-context.test.mjs` prüft bewusst Metadaten und Kontextdatei-Erzeugung aus lokaler Konfiguration. Das ist gut granuliert, weil dafür keine echten MCP-Server, Netzwerkzugriffe oder Daemons nötig sind.

## Schwächen und Risiken

1. **`mcp-cli.test.mjs` hängt an gebautem `dist/`**  
   Die Suite importiert bzw. startet `dist/bin/pibo.js`. Als schneller Entwicklungs-Subset ist sie nur zuverlässig, wenn vorher gebaut wurde. Ein lokaler Entwickler kann sonst veraltete Artefakte testen. Für Deployment-/Packaged-CLI-Validierung ist das sinnvoll, als Inner-Loop aber riskant.

2. **CLI-Parser-Fehlerpfade sind nur teilweise abgedeckt**  
   `src/mcp/index.ts` enthält viel Logik für unbekannte Optionen, fehlende Argumente, mehrdeutige Commands, `server/tool`-Schreibweise, `--config`, `--no-setup` und `grep`-/`call`-Argumente. Die bestehende Suite prüft vor allem Help und wenige Happy Paths. Parser-Regressionen würden oft erst in breiteren CLI-Flows auffallen.

3. **Tool-Filterung ist nicht isoliert getestet**  
   `src/mcp/config.ts` enthält mit `filterTools` und `isToolAllowed` eine sicherheitsrelevante Filterlogik (`allowedTools`, `disabledTools`, Glob-Matching, Disabled-vor-Allowed). Es gibt aktuell keinen kleinen Test, der diese Matrix ohne MCP-Verbindung abdeckt.

4. **`grep`-Pattern-Semantik ist ungeschützt**  
   `src/mcp/commands/grep.ts` exportiert `globToRegex`, aber die vorhandenen MCP-Tests prüfen keine Pattern-Fälle wie `*`, `?`, `**`, Slash-Grenzen oder Case-Insensitivity. Diese Funktion ist klein und ideal für eine schnelle Unit-Suite.

5. **Verbindungsbefehle werden nicht mit Fake-MCP-Server getestet**  
   `list`, `info`, `grep` und `call` sind entweder ungetestet oder nur über leere/fehlende Registry-Konfiguration abgesichert. Dadurch fehlen Tests für formatierten Tool-Output, Tool-not-found-Hinweise, `call`-JSON-Parsing und Fehler bei nicht erreichbaren Servern. Ein echter Server wäre zu breit; ein minimaler lokaler Fixture-Server wäre ein guter mittlerer Subset.

6. **Konfigurationspfad-Priorität fehlt als Testmatrix**  
   `src/mcp/config-command.ts` dokumentiert die Reihenfolge `-c/--config`, `MCP_CONFIG_PATH`, `./mcp_servers.json`, Home-Pfade. Die aktuellen Tests nutzen temporäre CWDs, aber sichern die Priorität nicht explizit ab. Gerade für Agentenläufe ist das relevant, weil Umgebungsvariablen häufig gesetzt sind.

## Fehlende oder anzupassende Tests

Empfohlene kleine Ergänzungen, ohne direkt eine große MCP-Integration-Suite zu bauen:

1. **Unit-Subset: MCP-Config-Filter**
   - Datei: neu oder erweitert, z. B. `test/mcp-config-filter.test.mjs`
   - Zielcode: `src/mcp/config.ts`
   - Fälle:
     - ohne Filter sind alle Tools erlaubt
     - `allowedTools: ["read_*"]` erlaubt nur passende Namen
     - `disabledTools` gewinnt gegen `allowedTools`
     - Glob-Matching ist case-insensitive
     - `?` matcht genau ein Zeichen

2. **Unit-Subset: MCP-Grep-Glob**
   - Datei: neu oder erweitert, z. B. `test/mcp-grep-pattern.test.mjs`
   - Zielcode: `src/mcp/commands/grep.ts`
   - Fälle:
     - `*file*` matcht Toolnamen ohne Slash-Grenzen
     - `server/*` matcht keine verschachtelten Slash-Namen
     - `server/**` matcht verschachtelte Namen
     - Sonderzeichen wie `.` werden escaped

3. **CLI-Parser-Negativpfade**
   - Datei: `test/mcp-cli.test.mjs` oder separates `test/mcp-cli-parser.test.mjs`
   - Zielcode: `src/mcp/index.ts`
   - Fälle über `execFile`:
     - `pibo mcp --unknown` endet mit `CLIENT_ERROR`
     - `pibo mcp call demo` meldet fehlendes Tool
     - `pibo mcp demo/tool` meldet mehrdeutigen Command und empfiehlt `call`/`info`
     - `pibo mcp grep` meldet fehlendes Pattern
     - `pibo mcp config add demo '{"command":"node"}' -c <path>` funktioniert auch mit Option nach den Positionsargumenten

4. **Kleiner Fake-Server-Subset für Discovery/Call**
   - Datei: z. B. `test/mcp-cli-fixture-server.test.mjs`
   - Scope: ein minimaler stdio-MCP-Server als Test-Fixture, kein Netzwerk, keine Daemon-Persistenz.
   - Fälle:
     - `pibo mcp` listet Tools
     - `pibo mcp info <server>` zeigt Parameter
     - `pibo mcp grep '*echo*'` findet den Fixture-Toolnamen
     - `pibo mcp call <server> echo '{"text":"hi"}'` extrahiert Textcontent korrekt

5. **Config-Pfad-Priorität**
   - Datei: `test/mcp-cli.test.mjs` oder separates `test/mcp-config-paths.test.mjs`
   - Ziel:
     - `-c` überschreibt `MCP_CONFIG_PATH`
     - `MCP_CONFIG_PATH` überschreibt CWD-Datei
     - `config path` gibt den tatsächlich gewählten Pfad aus

## Empfohlene granulare Test-Kommandos

Für schnelle Entwicklung am MCP-CLI-Bereich:

```bash
# Packaged-CLI-Smoke, braucht aktuelles dist/
node --test test/mcp-cli.test.mjs

# Agent-Kontext ohne MCP-Serververbindung
node --test test/mcp-agent-context.test.mjs

# Nach Ergänzung kleiner Unit-Tests
node --test test/mcp-config-filter.test.mjs
node --test test/mcp-grep-pattern.test.mjs
```

Vor Commit oder wenn `dist/`-Drift möglich ist:

```bash
npm run build
node --test test/mcp-cli.test.mjs test/mcp-agent-context.test.mjs
```

Breitere Validierung erst später im Flow:

```bash
npm run typecheck
npm test
```

## Konkrete nächste Schritte

1. Einen kleinen Unit-Test für `filterTools`/`isToolAllowed` hinzufügen. Das ist der beste erste Schritt, weil er schnell ist und sicherheitsrelevante Tool-Auswahl abdeckt.
2. `globToRegex` mit einer reinen Unit-Suite absichern. Das reduziert Risiko in `pibo mcp grep`, ohne MCP-Server starten zu müssen.
3. CLI-Negativpfade aus `src/mcp/index.ts` in 5-7 fokussierten `execFile`-Tests ergänzen.
4. Erst danach einen minimalen stdio-Fixture-Server für `list`/`info`/`grep`/`call` einführen. Dieser Test sollte separat bleiben, damit Entwickler weiterhin die reinen Parser-/Config-Subsets schnell laufen lassen können.
5. Mittelfristig `test/mcp-cli.test.mjs` klar als gebaute CLI-Smoke-Suite benennen oder durch ein zusätzliches `tsx src/bin/pibo.ts ...`-Subset ergänzen, damit Inner-Loop-Tests nicht versehentlich stale `dist/` validieren.
