## Codebase Patterns
- Security-boundary aggregate stories should pair visible scope-boundary copy with focused source coverage and targeted web-channel auth/redaction/inline-validation tests, including raw-XState rejection.
- Project workflow creation/start validation should resolve persisted UI-published Workflow IR from the registry/store; only static code projections should synthesize runnable definitions with the selected session profile.
- Completeness-contract aggregate stories can be verified with focused source-level coverage tests that bind existing source/API/browser surfaces to the contract before marking the aggregate story passing.
- For browser checks of Project human action controls, create/start a Project workflow session through the dev-auth API, seed persisted wait tokens in the container Project store, then open the Project session with `?view=workflow`.
- Builder validation/publish completeness can pair source-level UI panel assertions with targeted `web-channel` validate, before-publish rejection, and version-allocation flows.
- Workflow Builder coverage stories can pair a source-level UI/test matrix with targeted `web-channel` flows; compare layout-only changes by stripping `ui` metadata before checking runtime semantics.
- Workflow Library buttons and labels should consume source/status-derived API `actions`/`editability` metadata rather than re-deriving code/UI mutation rules in React rows.
- Rollout-gate docs should pair each phase with required automated test families and reference canonical product smoke checklists instead of duplicating behavior contracts.
- Run inspection checklist tests can combine a source-level UI coverage matrix with targeted `workflow-session-kind` and `web-channel` flows; seed missing registered human action refs to cover disabled controls and diagnostics.
- Workflow Builder publish panels should disable the Publish action from `validation.blocksPublish`/error diagnostics while leaving draft save and manual Validate available; the backend `before_publish` route remains the authoritative gate.
- Catalog missingRefs should filter to explicit missing/unavailable reference diagnostic codes; diagnostics that carry `registryRef` for params or kind mismatch context should stay in full diagnostics only.
- Workflow Builder manual Validate controls should call the draft validate route without `before_publish`; the publish route owns the `before_publish` trigger, while manual validation uses the route fallback/edit triggers.
- Broad V2 release-gate tests can assert a PRD coverage matrix across package tests, web-channel integration tests, and source-level UI surfaces instead of duplicating long flows.
- Traceability completeness stories can use a source-level doc parser test to bind spec requirements, task groups, and MUST checklist reviewability to the canonical PRD contract.
- Workflow lifecycle archive/delete UI should use in-page confirmation panels with copy that names default catalog/picker effects and historical Project snapshot behavior so browser checks can verify the safeguards.
- Workflow state mapping UI should write simple scoped `state.reads`/`state.writes` arrays through dropdown controls, trigger `state_edit` validation, and leave complex state DSL edits to raw Pibo Workflow IR.
- Project workflow create/start diagnostics should render sanitized diagnostic metadata (`code`, `path`, `nodeId`, `edgeId`, `registryRef`, `hint`) near the blocking action, not only in a global error banner.
- Dedicated workflow version-list/inspect APIs should derive visibility through `buildWorkflowCatalogList` before reading definitions so archive/delete filtering matches catalog and picker surfaces.
- Configured/not-started Project workflow UI should derive pre-run validation and blocking diagnostics from `workflowLifecycleEvents`, because no workflow run record or trace attempt exists before Start succeeds.
- Web-channel tests import `dist/`, so run `npm run build` before targeted `node --test test/web-channel.test.mjs` checks when server code changes.
- Raw JSON schema editors should write schema JSON back into Pibo Workflow IR and rely on the draft validation pipeline to surface `WorkflowInterfaceError.*` subset diagnostics rather than adding Zod, AJV, or form-builder layers.
- V2 workflow picker API options should carry a uniform `displayName`, optional description, and nullable `paramsSchema` even when the picker has richer metadata such as profile aliases, schemas, or workflow versions.
- V2 explicit non-goal browser checks should verify visible scope-boundary copy and scan interactive action labels, because explanatory panels intentionally mention deferred features without exposing them as actions.
- Project Workflow/XState navigation should derive nested workflow links from real `PiboWebSessionNode` child sessions and definition link/deleted state from Project session `workflowDefinitionLink` metadata.
- Registered guard/adapter params should flow from picker `paramsSchema` metadata; render params editors only for selected refs with schemas and reject params on refs without schemas.
- Workflow lifecycle observability for Chat Web belongs in durable `workflow_lifecycle_events` records and Project bootstrap data so configured-session run inspection can surface start-blocked diagnostics without a workflow run id.
- Project Workflow/XState run inspection panels should render status, current node, run history, node attempts, edge transfers, output, and error sections with explicit empty states for configured/not-started sessions while labeling XState as projection-only.
- Project workflow human action controls should enrich Project sessions from persisted wait-token records, list only registered action refs, and submit through Project workflow-session routes with sanitized diagnostics.
- Workflow Builder inspectors should patch the same draft Pibo Workflow IR through existing draft PATCH validation triggers; capability refs in inspectors should come from picker APIs instead of free-text fields.
- Nested workflow node UI should reuse the workflow-version picker API, store only `workflowId`/`workflowVersion` refs in parent graph IR, and navigate to a separate workflow viewer instead of inline-expanding child graphs.
- Workflow catalog persistence uses explicit `SqliteWorkflowRunStore` entities for UI identities, drafts, published versions, archive states, and delete tombstones; tombstones must not remove definition snapshots needed by historical run inspection.
- Workflow record source/status contracts should use exported value guards; persisted UI entity records assert `source: "ui"` while code workflows remain catalog projections.
- Workflow profile picker APIs should compose active custom Agent Designer profiles from `CustomAgentStore` with global `channelContext.getProfiles()` entries, while using archived custom profile rows only to produce diagnostics for already-referenced drafts.
- Workflow capability picker UI should use select-only registered refs, nullable metadata fields, and selected-ref diagnostics instead of manual text entry paths.
- Workflow Builder graph add-node defaults should derive registered refs from picker APIs instead of hard-coding fixture ids because dev/prod profile catalogs differ.
- Chat Web top-level product tabs need coordinated updates in `src/apps/chat-ui/src/App.tsx` for `ChatAppRoute`, navigation, and rendering, plus `src/apps/chat-ui/src/main.tsx` for URL parsing and the TanStack route tree.
- In the `pibo-dev-WorkflowsV2` browser worker, pass `--fresh-profile` to every `browser-use` command when the default `PIBo` Chrome profile is already locked by another browser process.
- When resolving PRD decision stories, update the focused PRD plus `README.md`, `tasks.md`, and `09-implementation-completeness-contract.md` if they repeat the same TBD or baseline contract.
- For V2 open-question stories, also clear matching stale open questions in the parent `spec.md` and `design.md` when the decision is now documented.
- Workflow Builder canvas state should remain a projection; persist saved layout through existing `WorkflowDefinition.ui.layout` and `WorkflowDefinition.ui.positions`.
- Prompt asset edits should create managed revisions and pin revision IDs/content hashes in published workflow versions and snapshots.
- V2 workflow HTTP contracts are documented under Chat Web routes: catalog/lifecycle APIs use `/api/chat/workflows`, while Project workflow session APIs use `/api/chat/projects/:projectId/workflow-sessions`.
- Workflow catalog list/inspect APIs should group static code projections, persisted UI drafts, and UI-published versions, while gating archived records behind `includeArchived=true`.
- V2 workflow catalog permissions treat UI-authored drafts and published versions as global authenticated resources; `ownerScope`/principal fields are audit/event metadata, not draft access gates.
- Use `createWorkflowDiagnosticReport` from `@pasko70/pibo-workflows` validation exports for V2 diagnostic grouping across Workflow Library, Builder, Project session creation, run start, and runtime failure surfaces.
- Normal UI workflow diagnostics should pass through Chat Web `sanitizeWorkflowDiagnostics` before persistence or response serialization so only code/message/severity/path/node/edge/registry ref/hint fields reach clients.
- Workflow session routing metadata uses the stable Pibo Session metadata key `workflowSessionKind` with values `main_workflow`, `nested_workflow`, `agent_node`, and `subagent`; Chat sidebar nodes expose it as `PiboWebSessionNode.workflowSessionKind`.
- Project workflow sidebars should build from Project root Pibo Sessions plus real parent/child Pibo Session descendants from `channelContext.findSessions`, not from logical workflow graph node ids.
- Published workflow versions use `hashWorkflowDefinition` plus `SqliteWorkflowRunStore.savePublishedWorkflowVersion`; saving the same body is idempotent, while changing a published `workflowId@version` body throws to preserve immutability.
- Workflow publish routes should allocate semantic versions inside a durable transaction, reserve both static catalog and persisted versions, and make repeated publish requests idempotent by draft id.
- Edit-published actions should use `POST /api/chat/workflows/:workflowId/drafts`, create or reuse the one active next-version draft for UI-authored published workflows, and keep code projections duplicate-only.
- Project workflow session creation should use the V2 `/api/chat/projects/:projectId/workflow-sessions` route and `workflow-versions` picker; keep legacy `/projects/:projectId/sessions` limited to `simple-chat`.
- Project workflow session records keep selected workflow metadata in `project_sessions.workflow_id`, `workflow_version`, and constrained lifecycle `state`; configured sessions must not set `workflow_run_id` until Start.
- Project workflow session update paths should whitelist presentation fields such as title/archive and reject workflow id/version/configuration changes; start must validate the stored snapshot matches the session selection.
- Chat Web API state that must survive multiple HTTP requests in tests should use durable storage; scaffold-only in-memory state needs a shared cache because the web-channel test harness may recreate `createChatWebApp` through `getWebApps`.
- V2 validation triggers should preserve non-validation builder diagnostics while replacing stale validation diagnostics for each draft load, edit, validate, publish, Project create, or start boundary.
- Chat Web workflow draft routes should persist UI drafts in a durable workflow registry/store table through `PiboDataStore` instead of module-level caches so drafts survive `createChatWebApp` recreation.
- Project workflow session creation should whitelist request fields, reject disallowed override families/arbitrary options before persistence, and store allowed input/prompt/model/thinking/fast config in `project_sessions.configuration_json` with explicit workflow/eligible-node scopes.
- Project view routing should combine `PiboWebSessionNode.workflowSessionKind` with `PiboProjectSession` workflow linkage; legacy `simple-chat` Project main sessions may carry `main_workflow` metadata but must still route to Terminal.
- V2 workflow security validation should reject inline executable fields and hidden LLM coercion at draft/publish/start boundaries; incompatible ports require visible registered adapter refs instead.
- Container web gateway restart commands should avoid `pgrep`/`pkill` patterns that match the wrapper shell; keep the Node process alive with a keepalive/nohup after `runWebGatewayServer` logs startup.
- Invalid raw Workflow IR text should be a transient warning diagnostic on the draft while preserving the last parsed `definition`; clear the warning after the next valid parsed raw/visual edit.
- Raw Workflow IR UI should submit `rawDefinitionText`, preserve invalid textarea content for repair, and render the last valid draft object separately from unsaved raw text.
- Project workflow configuration snapshots should be immutable Project-service records keyed by Pibo Session id; apply prompt overrides only to the effective snapshot definition while preserving the base definition and hash.
- Edge adapter UI should derive source output and target input ports from current Workflow IR; use-as-edge stores the target input as adapter output, while insert-node splits the edge through a visible deterministic adapter node.
- Workflow Library version history should consume all catalog states for visibility, while Project workflow-version pickers remain published-only selection surfaces.
- Workflow archive is a workflow-identity overlay; apply it consistently when projecting static catalog records, persisted versions, active drafts, inspect results, and picker options instead of mutating a single version.
- Project workflow start should validate the immutable session snapshot and claim the one allowed run atomically in `ChatProjectService`; duplicate starts return the existing run instead of revalidating live catalog state.
- Duplicate and edit-published draft creation should clone the resolved published Workflow IR and carry its definition hash; catalog summaries alone are not enough to preserve UI-authored graphs and layout.
- Workflow delete/tombstone state should be applied before archive state and filtered out of catalog, picker, duplicate, publish, archive, and Project creation surfaces; Project run links then fall back to snapshot-only definition-deleted state.
