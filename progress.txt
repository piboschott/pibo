## Codebase Patterns
- One-node agent workflow runs accept `initialGlobalState`; validate it with `validateWorkflowGlobalState`, render it through prompt state readers, and persist it in `WorkflowRun.state.global` via the workflow run store.
- Local node state is stored under `WorkflowRun.state.local[nodeId]`; dispatch helpers snapshot the current node state into `NodeAttempt.localState` at start and terminal/waiting boundaries so restart inspection can recover the current node state.
- Prompt workflow runtime coverage should include both one-node `promptTemplate` execution and graph-level `promptBuilder` execution after edge transfer so fixed and variable prompt paths stay covered.
- Agent `promptTemplate` rendering supports `{{input.path}}`, `{{global.path}}`, `{{state.global.path}}`, `{{state.path}}` as a global-state shortcut, and current-node `{{local.path}}` / `{{state.local.path}}`; unresolved placeholders are preserved in the final prompt.
- Agent `promptBuilder` refs should use `promptBuilderRef("id")` for registered TypeScript prompt builders; registry-backed validation checks `promptBuilders`, rejects unknown refs, and rejects nodes declaring both `promptTemplate` and `promptBuilder`.
- Agent `promptBuilder` execution requires passing a Workflow Registry with `promptBuilders` into `dispatchWorkflowAgentNode`/`runOneNodeAgentWorkflow`; runtime prompt builders receive scoped `global`, `local`, and optional `edgePayloads` readers before Pibo Runtime is called.
- Agent nodes record the final prompt under `nodeAttempt.metadata.finalPrompt` after prompt rendering/builder execution and before Pibo Runtime dispatch; keep prompt text out of workflow runtime events and persist attempts through `WorkflowNodeAttemptStore`.
- For runtime tests that must prove fixed Agent Designer profile selection, create `createWorkflowRegistry(workflowFixtureProviders)`, validate with `{ registry }`, and bridge to runtime with a `profileResolver` that calls `resolveWorkflowAgentProfile`.
- Registry-backed profile validation is opt-in through `validateWorkflow(definition, { registry })`; register fixed Agent Designer profiles with `registerWorkflowAgentProfile` or `WorkflowProviders.profiles` before validating agent-node workflows against a full registry.
- Agent runtime metadata should be created from resolved profile plus executor result: executor-returned effective tools/skills/context win, otherwise use `ResolvedAgentProfile` capability lists; always persist selected profile and effective routing metadata on the node attempt.
- Agent node runtime profile resolution uses the optional `profileResolver` hook before Pibo Runtime/session creation; the resolved profile id is passed to `agentExecutor`/`createPiboSessionRoutingAgentExecutor` and recorded in node metadata.
- Agent node V1 runtime policy is enforced by `validateWorkflow`: agent nodes must use `runtime: "pibo"` and a fixed Agent Designer profile (`fixedProfile("pibo-agent")` or `{ kind: "fixed", id }`).
- Visible adapter node dispatch goes through `dispatchWorkflowAdapterNode` with a registry-resolved TypeScript `adapterRef`; it validates node input/output, emits node lifecycle events, persists attempts, and records `metadata.adapterId`.
- Persist node attempts through `WorkflowNodeAttemptStore`/`SqliteWorkflowRunStore.saveNodeAttempt`; dispatch helpers should save attempts at start plus terminal/waiting/failure boundaries so restart inspection can recover node status by kind.
- Human node dispatch goes through `dispatchWorkflowHumanNode`; it validates node input, persists a `WorkflowWaitToken` through `WorkflowWaitTokenStore`, emits `wait.created`, and leaves the node/run in `waiting` with `metadata.waitTokenId`.
- Nested workflow node dispatch goes through `dispatchWorkflowNestedWorkflowNode` with a registry-resolved child workflow and injected `nestedWorkflowExecutor`; it validates parent/child input and child/node output, records `childRunId` metadata, and keeps parent state isolated from child state.
- Agent node dispatch goes through `dispatchWorkflowAgentNode` with an injected `agentExecutor`; it validates node input, routes through `createPiboSessionRoutingAgentExecutor`, validates node output, emits node lifecycle events, and records runtime/session metadata on the node attempt.
- Code node dispatch goes through `dispatchWorkflowCodeNode` with a `WorkflowRegistry`; registered handlers receive scoped `global`, `local`, `edge`, `emit`, and `command` context and state patches must be declared in `node.state.writes`.
- Registered edge adapter runtime tests should validate the workflow with a `WorkflowRegistry` and then call `transferWorkflowEdgeAdapterData` so both static adapter refs and runtime adapter output are covered.
- Use `transferWorkflowEdgeAdapterData` for registered edge-adapter runtime transfer; it resolves the adapter, validates source output, validates adapter output with `validateWorkflowEdgeAdapterOutput`, and only then returns `targetInput` for the downstream node.
- Registry-backed workflow validation should remain opt-in via `validateWorkflow(definition, { registry })` until each ref family is implemented, preserving schema/graph-only validation for fixtures without providers.
- Edge adapters should validate both structural adapter refs (`adapterRef(...)`) and declared adapter output compatibility with the target node input; actual registry resolution remains a separate validation layer.
- Use `adapterRef(...)` for registered deterministic TypeScript adapters in workflow IR; pass those refs to `edgeAdapter(...)` for edge-local adapters and to visible `adapter` node `handler` fields.
- Two-node direct-edge tests should validate the workflow definition, call `recordWorkflowEdgeTransfer`, then use `transferResult.targetInput` as the downstream node attempt input.
- Use `recordWorkflowEdgeTransfer` when a direct edge transfer should update the run cursor, emit `edge.transferred`, and persist the run; keep `transferWorkflowEdgeData` pure for validation/payload creation.
- Direct edge runtime transfer is exposed as `transferWorkflowEdgeData`; it creates an immutable `EdgeTransfer` from a completed source attempt output only after validating source output and target input ports.
- Direct workflow edge compatibility is conservative: text ports match text ports, and JSON ports match only when their semantic schema contracts are equivalent; otherwise use an explicit edge adapter or adapter node.
- Single-prompt one-node runtime coverage should assert validation, routed Pibo session creation/message emission, trace events, and reopened `SqliteWorkflowRunStore` facts in one end-to-end test.
- Workflow run persistence is exposed through `WorkflowRunStore` and `SqliteWorkflowRunStore` in `packages/workflows/src/store/index.ts`; inject a store into `runOneNodeAgentWorkflow` when a runtime path must persist run status/cursor/input/output facts.
- One-node agent runtime trace/event projection uses the optional `emitEvent` hook on `runOneNodeAgentWorkflow`; it receives the same `WorkflowRuntimeEvent` objects returned in `result.events`.
- One-node agent runtime execution is exposed from `packages/workflows/src/runtime/index.ts` as `runOneNodeAgentWorkflow`; use `createPiboSessionRoutingAgentExecutor` to adapt a Pibo channel/session routing surface (`createSession`, `emit`, `subscribe`) into the injected `agentExecutor` seam.
- Use `validateWorkflowInput`, `validateNodeOutput`, and `validateWorkflowOutput` at runtime boundaries; they all delegate to shared port/schema value validation and emit JSONPath-like `WorkflowInterfaceError.*` diagnostics.
- Call `validateWorkflowInput(definition, input)` before starting workflow execution; it validates text ports as strings and JSON ports against the V1 schema subset with JSONPath-like diagnostics.
- Workflow validation code should emit `WorkflowDiagnostic` objects with `WorkflowInterfaceError.*` codes and JSONPath-like `path` values instead of throwing, then aggregate through `ValidationResult`.
- Workflow graph validation diagnostics use the `WorkflowGraphError.*` code family and should include `edgeId`, missing `nodeId`, and JSONPath-like paths for source/target refs.
- Workflow package tests live under `packages/workflows/src/testing/*.test.ts` and run with `npm --prefix packages/workflows test` (`tsx --test`).
- Workflow-system implementation progress is currently tracked in `docs/specs/changes/pibo-workflow-system-v1/tasks.md`; the referenced `prd_*.json` files are not present in this worktree.
- Keep the new workflow framework isolated under `packages/workflows` so future iterations can evolve the package without touching the host gateway or existing app surfaces.
- Export public workflow package types through `packages/workflows/src/index.ts` using `.js` module specifiers because the package uses `moduleResolution: NodeNext`.
- Keep reusable workflow examples in `packages/workflows/src/fixtures/index.ts`; pair registry-ref-heavy fixtures with `workflowFixtureProviders` so future tests can resolve handlers/adapters/guards without embedding closures in definitions.
- Put public workflow authoring helpers in `packages/workflows/src/api/index.ts`, export them from the package root, and test them through `../index.js`.
- Treat `docs/specs/changes/pibo-workflow-system-v1/structured-outputs-json-schema-subset.md` as the canonical V1 JSON port schema subset doc; keep it aligned with `packages/workflows/src/validation/index.ts`.
