## Codebase Patterns
- Use `recordWorkflowEdgeTransfer` when a direct edge transfer should update the run cursor, emit `edge.transferred`, and persist the run; keep `transferWorkflowEdgeData` pure for validation/payload creation.
- Direct edge runtime transfer is exposed as `transferWorkflowEdgeData`; it creates an immutable `EdgeTransfer` from a completed source attempt output only after validating source output and target input ports.
- Direct workflow edge compatibility is conservative: text ports match text ports, and JSON ports match only when their semantic schema contracts are equivalent; otherwise use an explicit edge adapter or adapter node.
- Single-prompt one-node runtime coverage should assert validation, routed Pibo session creation/message emission, trace events, and reopened `SqliteWorkflowRunStore` facts in one end-to-end test.
- Workflow run persistence is exposed through `WorkflowRunStore` and `SqliteWorkflowRunStore` in `packages/workflows/src/store/index.ts`; inject a store into `runOneNodeAgentWorkflow` when a runtime path must persist run status/cursor/input/output facts.
- One-node agent runtime trace/event projection uses the optional `emitEvent` hook on `runOneNodeAgentWorkflow`; it receives the same `WorkflowRuntimeEvent` objects returned in `result.events`.
- One-node agent runtime execution is exposed from `packages/workflows/src/runtime/index.ts` as `runOneNodeAgentWorkflow`; use `createPiboSessionRoutingAgentExecutor` to adapt a Pibo channel/session routing surface (`createSession`, `emit`, `subscribe`) into the injected `agentExecutor` seam.
- Use `validateWorkflowInput`, `validateNodeOutput`, and `validateWorkflowOutput` at runtime boundaries; they all delegate to shared port/schema value validation and emit JSONPath-like `WorkflowInterfaceError.*` diagnostics.
- Call `validateWorkflowInput(definition, input)` before starting workflow execution; it validates text ports as strings and JSON ports against the V1 schema subset with JSONPath-like diagnostics.
- Workflow validation code should emit `WorkflowDiagnostic` objects with `WorkflowInterfaceError.*` codes and JSONPath-like `path` values instead of throwing, then aggregate through `ValidationResult`.
- Workflow graph validation diagnostics use the `WorkflowGraphError.*` code family and should include `edgeId`, missing `nodeId`, and JSONPath-like paths for source/target refs.
- Workflow package tests live under `packages/workflows/src/testing/*.test.ts` and run with `npm --prefix packages/workflows test` (`tsx --test`).
- Workflow-system implementation progress is currently tracked in `docs/specs/changes/pibo-workflow-system-v1/tasks.md`; the referenced `prd_*.json` files are not present in this worktree.
- Keep the new workflow framework isolated under `packages/workflows` so future iterations can evolve the package without touching the host gateway or existing app surfaces.
- Export public workflow package types through `packages/workflows/src/index.ts` using `.js` module specifiers because the package uses `moduleResolution: NodeNext`.
- Keep reusable workflow examples in `packages/workflows/src/fixtures/index.ts`; pair registry-ref-heavy fixtures with `workflowFixtureProviders` so future tests can resolve handlers/adapters/guards without embedding closures in definitions.
- Put public workflow authoring helpers in `packages/workflows/src/api/index.ts`, export them from the package root, and test them through `../index.js`.
- Treat `docs/specs/changes/pibo-workflow-system-v1/structured-outputs-json-schema-subset.md` as the canonical V1 JSON port schema subset doc; keep it aligned with `packages/workflows/src/validation/index.ts`.
