## Codebase Patterns
- Workflow validation code should emit `WorkflowDiagnostic` objects with `WorkflowInterfaceError.*` codes and JSONPath-like `path` values instead of throwing, then aggregate through `ValidationResult`.
- Workflow package tests live under `packages/workflows/src/testing/*.test.ts` and run with `npm --prefix packages/workflows test` (`tsx --test`).
- Workflow-system implementation progress is currently tracked in `docs/specs/changes/pibo-workflow-system-v1/tasks.md`; the referenced `prd_*.json` files are not present in this worktree.
- Keep the new workflow framework isolated under `packages/workflows` so future iterations can evolve the package without touching the host gateway or existing app surfaces.
- Export public workflow package types through `packages/workflows/src/index.ts` using `.js` module specifiers because the package uses `moduleResolution: NodeNext`.
- Keep reusable workflow examples in `packages/workflows/src/fixtures/index.ts`; pair registry-ref-heavy fixtures with `workflowFixtureProviders` so future tests can resolve handlers/adapters/guards without embedding closures in definitions.
